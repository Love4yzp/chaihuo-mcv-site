# 设计系统 + AI 可理解规范 — 设计文档

**日期**: 2026-05-31
**项目**: 柴火基地车官网 (Chaihuo MCV Site)
**状态**: 已通过 brainstorming,待落实施计划

---

## 1. 背景与目标

### 真实痛点

站点作者**不是设计师、也不是开发者**,在开发过程中缺少一套「好的 design」。
直接后果:每加一个功能都靠临时拍脑袋,产出不一致;也无法稳定地指挥 AI 产出"在同一套语言里"的结果。

### 这件事到底要解决什么

不是「换文件夹组织方式」,而是:

> **给站点建立一套设计语言 + 让 AI 每次都能稳定地理解并遵守它,同时在开发全新功能时仍有自主发挥空间。**

### 关键洞察:为什么不是「一本钉死的组件目录」

传统设计系统是「组件清单」,把每个东西长什么样钉死。
这种做法碰到**全新功能**(如地图)就失效,甚至碍事——地图的缩放、标签排布、轨迹绘制是它独有的发明,没有任何组件清单能预先规定。

因此设计系统必须**分层**,明确告诉 AI:哪些必须守、哪些要领会精神、哪些可以自由发挥。

---

## 2. 核心方案:三层设计系统

```
🔒 第一层 · 不可破的硬约束 (Invariants)
   颜色令牌、字号阶梯、间距节奏、圆角、品牌色含义、
   无障碍 (reduced-motion / 对比度)、中英文对等
   → 任何功能(含地图)都必须用这些。

🧭 第二层 · 要领会的精神 (Principles)
   品牌气质、「探险黄 + 远征探索 + 手绘感 + Premium Restrained 克制动效」、
   动效哲学、何时克制何时活泼
   → 给【全新功能】用。精神能迁移,组件规格不能迁移。

🎨 第三层 · 自由发挥区 (Open / 仅参考)
   具体组合方式、新交互、新布局(在栅格内)
   → 文档只记「我们一般这么做」,标注【参考,非强制】。功能需要时大胆偏离。
```

### 用地图功能验证(成立性检查)

- 🔒 **硬约束**:地图的城市标签、面板照样用 `brand` 品牌色、同一套字号阶梯、同样间距 → 再特殊也不脱离整站。
- 🧭 **精神**:缩放与手绘风格呼应「远征探索」调性、尊重 `reduced-motion`(近期 commit 刚做过) → 文档给方向不给规格。
- 🎨 **自由**:`label-layout.ts` 标签排布、轨迹绘制、缩放交互 → 地图独有发明,文档不规定,只要落在前两层内。

结论:分层对地图完全成立。AI 拿到新功能读的是「哪些线不能碰、要延续什么气质」,而非「照抄第 N 号卡片」。

## 3. 现状盘点(已有 vs 缺口)

### 已有(主要在 `src/styles/theme.css`,质量不错)

- **颜色**:品牌色「探险黄 `#f3d230`」+ dark/hover/light/foreground 四变体;中性色阶 950→50;surface 令牌;明确的 **60-30-10 配色法则**。
- **排版**:h1–h4 字号/字重/行高/字距已定死(h1 700 字重 + 紧字距);用 Tailwind 默认字号阶梯。
- **圆角**:`--radius: 0.625rem` + sm/md/lg/xl。
- **动效签名**:页面切换 `250ms 淡出 / 350ms 淡入+上滑`,统一缓动 `cubic-bezier(0.4,0,0.2,1)`,注释「Premium Restrained」。
- **AI 上下文**:`CLAUDE.md` 已较完善(含「用 `text-brand` 等、不要硬编码 hex / 不要 gray-xxx」)。

### 缺口

1. `fonts.css` 为空(字体在别处加载)——需在 DESIGN.md 记录字体实际来源。
2. 没有任何地方写「🧭 精神层」。
3. 没有 `/elements` 可视化参考页。
4. `CLAUDE.md` 未把设计系统三层规则收进去。
5. 无 linter / formatter。

> 含义:**硬约束层主要是「把已有的文档化」**,真正的新活儿是补精神层 + 建可视化页 + 接进 CLAUDE.md + 轻量工程化。

## 4. 交付物

### ① `docs/DESIGN.md` —— 给 AI 的「自由度说明书」

按三层撰写,每条标注档位:

- **🔒 不可破**:引用 `theme.css` 令牌(颜色/字号/圆角/间距节奏)、60-30-10 法则、无障碍底线(`reduced-motion`、对比度)、中英文必须对等。措辞「永远用这些,不要硬编码 hex / 不要 gray-xxx」。
- **🧭 要领会的精神**:把「探险黄 + 远征探索 + 手绘感 + Premium Restrained 克制动效」写成 4–6 条气质原则,明确这是给新功能延续"感觉"用的。
- **🎨 自由区**:列已有常见模式(卡片、手风琴、时间线、轮播)作参考,标注「非强制,新功能可大胆偏离,只要落在前两层内」。

### ② `/elements` 可视化参考页(中英各一)

- 真实页面,摆出色板、字号阶梯、按钮/徽章/卡片/手风琴、动效示例。
- 价值:作者看不懂代码也能用眼睛检查整套系统,以后能指着说「就要这个感觉」。
- 实现:React island,复用现有 `src/app/components/ui/` 组件,**零新依赖**;走现有 i18n(`src/pages/` + `src/pages/en/`)。

### ③ 更新 `CLAUDE.md`

新增「设计系统」一节:摘要三层档位 + 指向 `docs/DESIGN.md` 与 `/elements`。
目的:AI 每次开新功能先读三档,即知「哪条线不能碰、延续什么气质、哪里自由发挥」。

### ④ 工程化(轻量)

引入 **Biome** 做 lint/format(现无 linter),接进现有 `pnpm check`。
**只统一格式,不引入重型/激进规则。**

## 5. 分阶段交付计划

每步完成后跑一次 `pnpm harness`(`astro check` + 内容校验 + Playwright smoke/visual/ui-audit)验证。

| 期 | 内容 | 风险 | 验证 |
|---|---|---|---|
| 1 | 写 `docs/DESIGN.md`(三层) | 纯文档,零风险 | 人工通读 |
| 2 | 建 `/elements` 页(zh/en) | 低,新增页面不动旧的 | `pnpm harness` |
| 3 | 更新 `CLAUDE.md` 设计系统节 | 纯文档 | 人工通读 |
| 4 | 接入 Biome | 低,只动格式 | `pnpm check` 通过 |

**期 5(目录重组 feature-first / 方案 C):本次不做**,留待后续单独评估。

---

## 6. 范围边界 (YAGNI)

- **不做**目录重组(原方案 C)。
- **不**引入运行时主题切换之外的新设计依赖;`/elements` 复用现有 `ui/` 组件。
- **不**改动 `src/app/components/ui/`(shadcn 原语,按约定不手改)。
- **不**引入激进 lint 规则,Biome 仅做格式统一。
- 设计系统**不**写成穷举组件目录——只写三层档位 + 参考模式。

---

## 7. 成功标准

1. `docs/DESIGN.md` 存在,清晰分出 🔒/🧭/🎨 三档,硬约束与 `theme.css` 一致。
2. `/elements` 中英页可访问,完整展示色板/字号/组件/动效,`pnpm harness` 通过。
3. `CLAUDE.md` 含设计系统节并指向上述两者。
4. Biome 接入,`pnpm check` 通过且不破坏现有构建。
5. 验收检验:给 AI 一个「开发全新功能(如新地图类模块)」的任务,它能正确引用三层(守硬约束、延续精神、自由组合),而非套用固定组件。
